Foundations

Git vs GitHub — they are not the same thing

This is the #1 confusion for beginners. Git and GitHub are two separate things that happen to work beautifully together.

Git

A tool installed on your computer. It tracks every change you make to your files, lets you save snapshots (commits), and manages versions — entirely offline. Think of it as a time machine for your code.

GitHub

A website that stores your Git history in the cloud. It's a remote backup of your project, plus collaboration tools like Issues, Pull Requests, and Actions. You push your local Git history up to GitHub.

💡
Git is the engine. GitHub is the garage where you park it. You can use Git with no internet — you just can't share or backup until you push to GitHub.

What you've been doing vs what you should do

What you do now

Zip folder → upload to GitHub. This is basically just file storage. GitHub can't see your history, you can't rollback, you can't branch.

What Git gives you

Every meaningful change is a commit. GitHub stores all of them. You can jump to any commit in history, branch off to experiment, and merge it back.

Foundations

Core concepts — the vocabulary

Git has 6 terms you absolutely must know. Everything else is built on these.

Repository (repo)

Your project folder, but with a hidden .git folder inside it. That hidden folder is Git's entire database — your full history. When you push to GitHub, this database goes with it.

Commit

A saved snapshot of your project at a moment in time. A commit has: the changes you made, a message describing them, and a unique ID (like a3f8c12). Your project history is a chain of commits.

Commit 1Initial setup
→
Commit 2Add login page
→
Commit 3Fix button bug
→
HEADWhere you are now

Branch

A parallel version of your project. Your main code lives on the main branch. You can create a new branch to try something — your main code is untouched until you merge.

Staging area

A "waiting room" for your changes. You git add files to stage them, then git commit saves everything staged. This lets you pick exactly which changes go into a commit.

Push / Pull

Push = send your local commits to GitHub. Pull = download commits from GitHub to your machine. For solo work, you push to back up and pull on another device.

Merge

Combining a branch back into main. Git is smart about merging — it only looks at what changed, line by line.

First-time setup

Do this once on your machine. You only need to do this one time ever.

Step 1 — Install Git

Download from git-scm.com. On Mac you may already have it. Run this to check:

Terminal
git --version

Step 2 — Tell Git who you are

Git stamps every commit with your name and email. Use the same email as your GitHub account.

Terminal
git config --global user.name "Your Name"
git config --global user.email "you@email.com"

Step 3 — Authenticate with GitHub

Install the GitHub CLI — it handles login securely so you don't have to deal with SSH keys manually.

Terminal
gh auth login

Follow the prompts. Choose HTTPS, and authenticate via browser. Once done, your machine can push and pull from GitHub without a password every time.

✓
After setup, run git config --list to verify your name and email are saved correctly.

Step 4 — Set default branch name

Terminal
git config --global init.defaultBranch main
Daily Use

Your first repo — two ways to start

Every project starts one of two ways. Pick based on where your code currently lives.

Option A — Existing project on your computer

Navigate into your project folder in the terminal, then:

Terminal — inside your project folder
git init                     # creates the hidden .git folder
git add .                    # stage all your current files
git commit -m "Initial commit"

Then create a repo on GitHub (github.com → New repository), and link it:

Terminal
git remote add origin https://github.com/yourname/repo-name.git
git push -u origin main

Option B — Start fresh from GitHub (recommended)

Create the repo on GitHub first, then clone it to your machine. This is the cleanest approach.

Terminal
gh repo create my-app --public --clone
cd my-app

Or using the website: create repo → copy clone URL → then:

Terminal
git clone https://github.com/yourname/my-app.git
cd my-app
💡
The -u origin main flag in the first push sets the "upstream" — from that point on, just git push is enough, no need to specify every time.

The commit loop — your daily rhythm

This is the loop you'll repeat dozens of times per day. Internalize this and you'll feel like a real developer.

1
Write or edit code
2
Check what changed
3
Stage the changes you want to commit
4
Write a commit message and commit
5
Push to GitHub when you want to back up
The full loop in commands
git status                   # see what files changed
git diff                     # see exact line changes
git add index.html           # stage a specific file
git add .                    # OR stage everything changed
git commit -m "Add navbar component"
git push                     # send to GitHub

Writing good commit messages

Use the format: verb + what you did. Start with a capital letter. Keep it under 72 characters.

✓ Good messages

Add login page
Fix crash on empty input
Update Tally dashboard layout
Remove unused imports

✗ Bad messages

changes
fix
asdfasdf
updated stuff again

💡
Commit early, commit often. A commit is free. Committing every 20–30 minutes of meaningful work is healthy. You can always look back and see exactly what you did and why.
Daily Use

Push & pull — syncing with GitHub

How your local commits travel to the cloud, and how you bring remote changes down.

Pushing your work to GitHub

After one or more commits, push to GitHub to back them up:

Terminal
git push

That's it. The -u origin main you set on the first push means future pushes are just git push.

Pulling changes from GitHub

If you work on multiple machines, or if you made a change directly on GitHub (like editing README), you need to pull before working:

Terminal
git pull
⚠
Always pull before you start working if there could be remote changes. This prevents merge conflicts from building up.

Checking your status at any time

Terminal
git status                          # what's staged, unstaged, untracked
git log                             # full commit history
git log --oneline                   # compact history (much nicer)
git log --oneline --graph --all     # visual branch history

Seeing what's on GitHub vs local

Terminal
git fetch              # download remote info without merging
git status             # now shows "Your branch is X commits behind"
💡
git fetch is "check what's new". git pull is "check what's new AND apply it". Use fetch when you want to look before you merge.
Branching

What is a branch — visually

A branch is a named pointer to a commit. When you branch off, you're creating a diverging timeline — your main code stays untouched while you work.

main: A ── B ── C ──────────────── G \ / feature: D ── E ── F ──'

A, B, C are commits on main. You branched off at C, made commits D, E, F on your feature branch. When ready, you merge back into G on main. The feature branch's work is now part of main.

Creating and switching branches

Terminal
git branch feature/login      # create a branch
git switch feature/login      # switch to it

# Or do both in one command (most common):
git switch -c feature/login

Listing branches

Terminal
git branch                   # local branches
git branch -a                # local + remote branches

The HEAD pointer

HEAD is just Git's way of saying "where am I right now." When you switch branches, HEAD moves. When you commit, HEAD advances to the new commit.

💡
Branch names are just labels. Use feature/, fix/, or experiment/ as prefixes. Example: feature/tally-categories, fix/login-crash.
Branching

When to branch — the decision rule

For solo projects, a simple rule: branch whenever you're working on something that could break what's already working.

Branch for this

New feature that takes more than 1 session

Experimenting with a different approach

Fixing a bug that requires touching multiple files

Redesigning a component

Trying a new library or architecture

Don't need to branch

Tiny text or style fix

Updating README

Quick one-line bug fix

Adding a comment or cleaning up whitespace

A complete branch workflow

Terminal
git switch main                      # always start from main
git pull                             # make sure it's up to date

git switch -c feature/dashboard      # create + switch

# ... do your work, make commits ...
git add .
git commit -m "Add dashboard screen"

git push -u origin feature/dashboard # push the new branch

# when done, merge back into main:
git switch main
git merge feature/dashboard
git push

git branch -d feature/dashboard      # delete (optional)
✦
For Tally: create a branch for each screen you build. feature/transactions-screen, feature/budget-categories, etc. Each branch is one unit of work.
Branching

Merging & conflicts

How branches come back together — and what to do when Git can't figure it out on its own.

Fast-forward merge (the easy case)

If main hasn't moved since you branched, Git simply slides main forward to your branch tip. No merge commit, no conflict — it just works.

Before: main → C, feature → C → D → E After: main → E (fast-forwarded)

Merge commit (when both branches moved)

If main got new commits while you worked on your branch, Git creates a "merge commit" that ties both timelines together. Usually automatic.

Merge conflicts

A conflict happens when two branches edited the same line differently. Git can't decide which is right — it asks you.

What a conflict looks like in a file
<<<<<<< HEAD (your branch)
backgroundColor: '#FF6B35',
=======
backgroundColor: '#2D3436',
>>>>>>> main

You manually edit the file to keep what you want (delete the markers), then:

Terminal
git add the-conflicted-file.js
git commit -m "Resolve merge conflict in theme colors"
⚠
VS Code has a built-in merge conflict UI — you'll see "Accept Current Change / Accept Incoming" buttons. Use those, it's much easier than editing raw markers.
Recovery

Undo & rollback — your safety net

This is where Git pays off. You can undo almost anything. Here are the scenarios from safest to most drastic.

Undo staged changes (before committing)

Terminal
git restore --staged index.html   # unstage (keep your edits)
git restore index.html            # discard edits entirely (gone!)
⛔
git restore without --staged permanently discards your local changes. No undo for this one.

Undo the last commit (keep your changes)

Terminal
git reset --soft HEAD~1    # uncommit, keep files staged
git reset HEAD~1           # uncommit, keep files unstaged

View history and jump to a past commit

Terminal
git log --oneline          # find the commit ID you want
git checkout a3f8c12       # travel to that commit (read-only)
git switch main            # come back to the present

Revert a commit (safe for shared repos)

Creates a new commit that undoes a past one. Doesn't rewrite history — safe to use even after pushing.

Terminal
git revert a3f8c12         # creates an undo commit

Hard reset (nuclear — destroys history after it)

Terminal
git reset --hard a3f8c12   # ⚠ erases commits after this point
⚠
If you've already pushed to GitHub, use git revert (safe) not git reset --hard (rewrites history). Rewriting pushed history causes problems.
GitHub Features

.gitignore — what not to track

Some files should never go into Git: build output, secret keys, dependencies, OS junk. The .gitignore file tells Git to skip them entirely.

Create a .gitignore at your project root

.gitignore
# Node / JavaScript
node_modules/
dist/
.env
.env.local

# React Native / Expo
.expo/
android/
ios/

# OS files
.DS_Store
Thumbs.db

# IDE
.vscode/settings.json
.idea/
⛔
Never commit your .env file. It contains API keys and secrets. Add it to .gitignore from day one. If you accidentally pushed it — change those keys immediately, they're compromised.

Quick way — use GitHub's built-in templates

When creating a repo on GitHub, select "Add .gitignore" and choose your framework (Node, React Native, etc.). GitHub generates the right one for you.

If you already committed something you shouldn't have

Terminal
git rm --cached .env            # stop tracking (keeps local file)
echo ".env" >> .gitignore       # add to gitignore
git commit -m "Remove .env from tracking"

README & Issues — GitHub as a project tool

GitHub isn't just a code host. It's a complete project management tool for solo developers.

README.md

The README.md is the homepage of your GitHub repo. Write it in Markdown. For Tally and your other apps, use it as your living project doc.

README.md structure for a personal project
# Tally — Personal Finance Tracker

A local-first offline finance tracker for Indian urban consumers.

## Tech Stack
- React Native + Expo
- SQLite (local-first)
- React Query

## Running locally
1. Clone the repo
2. Run `npm install`
3. Run `npx expo start`

## Status: In development

Issues — your personal task tracker

GitHub Issues are free and live right next to your code. Use them as your to-do list for the project.

How to use Issues

Go to your repo → Issues → New Issue. Create one for every feature, bug, or improvement. Close it when done.

Linking commits to issues

In a commit message: Fix login crash (closes #12) — GitHub auto-closes Issue #12 when that commit is pushed to main.

GitHub Projects (Kanban board)

Under the Projects tab, create a board (To Do → In Progress → Done) and link Issues to it. For solo dev on Tally, this is a lightweight project manager with no extra tools needed.

GitHub Features

GitHub Actions — automation for free

GitHub Actions lets you run code automatically when you push. The most useful thing for solo devs: auto-linting and auto-deploy on every push.

How it works

You write a YAML file in .github/workflows/. GitHub reads it and runs whatever you specify — on every push, PR, or on a schedule.

.github/workflows/lint.yml — run ESLint on every push
name: Lint

on: [push]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm install
      - run: npm run lint

Push this file, and GitHub will run your linter every time you push. You'll see a green ✓ or red ✗ on every commit in GitHub.

Other useful Actions for personal projects

Auto-deploy

Deploy to Vercel, Netlify, or Expo on every push to main. Most platforms have official Actions.

Run tests

If you write tests (Jest, etc.), run them on every push so you catch regressions instantly.

Your Workflow

Your complete daily workflow

Here's exactly how to use Git for every project — Tally or anything else. Follow this and you'll build real developer habits fast.

Starting a work session

Terminal
cd tally                     # go to your project
git pull                     # sync with GitHub
git log --oneline -5         # remind yourself where you left off

Working on a new feature

Terminal
git switch -c feature/budget-screen
# ... write code ...
git status
git add .
git commit -m "Add budget categories screen"
git push -u origin feature/budget-screen

Done with the feature — merge it

Terminal
git switch main
git pull
git merge feature/budget-screen
git push
git branch -d feature/budget-screen

Quick fixes — straight to main

Terminal
git switch main
# fix the thing
git add .
git commit -m "Fix typo in onboarding copy"
git push
✦
Rule of thumb: if a feature takes more than one sitting to build, give it a branch. If it takes 10 minutes, commit to main.

At the end of every session

Terminal
git status                   # make sure nothing is uncommitted
git push                     # back up everything to GitHub
💡
Make it a habit: before you close your laptop, git status → git push. You'll never lose work again.
